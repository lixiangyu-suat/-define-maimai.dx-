在算法竞赛和工程实践中，DFS（深度优先搜索）根据**状态维护方式**和**搜索目标**的不同，通常可以归纳为四种核心框架。你提到的 `swap` 和“记录 9 位数”实际上代表了处理“状态”的两种主流流派。

---

## 1. 现场恢复流 (Backtracking / 状态回溯)

这是你刚才在骑士问题中看到的写法。其核心思想是：**全场只有一个状态实例（通常是全局变量或引用传递）**。

- **操作逻辑**：修改状态 $\rightarrow$ 递归 $\rightarrow$ **改回原样**。
    
- **适用场景**：状态对象很大（如 $20 \times 20$ 的棋盘），每次复制状态开销太高。
### 剪枝：约束限制流 (Bounded DFS / IDDFS & IDA*)

这就是我们之前讨论的，为了在 DFS 中寻找“最短路”而生的框架。

- **操作逻辑**：外层套一个 `limit` 循环，内层 DFS 配合剪枝。
    
- **适用场景**：状态空间巨大无法 BFS，但目标深度较浅的情况。
    
-  例题： [[骑士精神]]
- 优点：极度节省内存，空间复杂度仅为 $O(depth)$。
- 缺点：仅仅适用于下一层“分裂”式扩展，如树；如果边太多，会退化为暴力dfs；


## 2. 状态传递流 (State Passing / 显式状态)

在 8 数码或哈夫曼编码中常见。将当前棋盘压成一个 `string` 或 `long long`。

- **操作逻辑**：计算出下一个状态的新值，并作为**形参**传给下一层。
    
- **适用场景**：状态容易压缩（如 9 位数），或者需要配合 `std::unordered_set` 进行全局判重。
    
- **优点**：逻辑清晰，不需要回溯（因为每一层都有自己的 `copy`），天生支持判重。
    
- **缺点**：频繁创建对象和哈希查找会带来额外的时间和内存开销。
    

## 3. 记忆化搜索 (Memoization / DFS + 动态规划)

当你发现“同一个状态可能从不同路径被访问到”时使用。

- **操作逻辑**：进入 DFS 先查表 $\rightarrow$ 有结果直接返回 $\rightarrow$ 没结果则计算并存入表。
    
- **适用场景**：存在大量重叠子问题（如数位 DP、区间 DP）。
    
- **核心**：它让搜索树变成了图，将指数级复杂度降为多项式级。
    


---

### DFS 框架核心总结表

|**框架类型**|**核心操作**|**状态存储**|**判重方式**|
|---|---|---|---|
|**回溯法**|`Modify` -> `DFS` -> `Restore`|全局/单实例|`last_move` 或 路径标记|
|**显式传递**|`DFS(NextState)`|局部变量/副本|`std::set` / `Hash`|
|**记忆化**|`if(memo[s]) return`|全局 Lookup Table|查表即判重|
|**迭代加深**|`if(g + h > limit) return`|同回溯法|估价函数剪枝|

    if (is_goal()) { res = min(res, step); return; }
    for (auto& dir : dirs) {
        int nx = x + dir.x, ny = y + dir.y;
        if (!is_valid(nx, ny)) continue;

        swap(board[x][y], board[nx][ny]);

        swap(board[x][y], board[nx][ny]); 
    }
}

bool dfs(int g) {
    int h = evaluate();

## 4. 注意事项

- **越界检查**：永远先检查坐标合法性，再访问数组。
    
- **位运算优先级**：`&`, `|`, `^` 优先级低于 `==` 和 `+`，务必加括号。
    
- **栈溢出**：DFS 深度超过 $10^5$ 时需警惕系统栈爆掉，考虑手动模拟栈或改用 BFS。
